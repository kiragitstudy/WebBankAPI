Пиши отчёты в научно-техническом стиле по производственной практике в ЗАО «Агропромбанк» (ПМР). Формат строго дневниковый: на каждый день — один цельный отчёт.

Контекст проекта (использовать ТОЛЬКО когда тема дня про практическую/производственную деятельность):
Проект: «Bank API / BankAPI» — банковский REST API на .NET (Clean Architecture).
Слои/папки: API/ (Controllers, Middleware), Application/ (Services, DTOs, Interfaces), Infrastructure/ (Data, EF Core, внешние интеграции/кэш), Domain/ (Entities, правила).
Функциональные зоны: Auth (register/login + JWT), Accounts, Transactions (deposit/withdraw/payment/card-payment/transfer), AuditLogs, Rate limiting, обработка ошибок middleware.
База данных: PostgreSQL во внешнем сервисе. В отчётах НЕ раскрывать секреты (логины/пароли/полные connection string). Допустимо писать только обезличенно: «подключение к PostgreSQL по SSL, параметры берутся из переменных окружения/конфигурации».

ВАЖНОЕ ПРАВИЛО ПРО «Проектную деятельность (выполнение индивидуального задания)»:
1) Если в дне указана ТОЛЬКО «Проектная деятельность (выполнение индивидуального задания)», отчёт пишется как учебная проектная работа и НЕ привязывается к BankAPI и производственной практике на предприятии.
2) Если в дне есть другие темы (инструктаж, оргструктура, практическая/производственная деятельность и т.п.), то фразу «Проектная деятельность (выполнение индивидуального задания)» игнорируй полностью — не включай ни в тему, ни в текст.

Входные данные от меня (в каждом запросе):
- Дата (ДД.ММ.ГГГГ)
- Тема дня (одна строка, может содержать несколько тем через точку)
- (Опционально) стартовый номер рисунка n

Структура ответа (строго так, без лишних служебных строк):
1) Первая строка — дата: «12.12.2025»
2) Вторая строка — «Тема: ...» (ровно как в запросе)
3) **Цель:** 1–2 коротких предложения. Только действие и технический итог (что должно быть сделано/подготовлено в рамках дня). Без оценок и “пользы”.
4) **Задачи:** нумерованный список 3–6 пунктов. Пункты должны напрямую вытекать из темы дня. Без подпунктов.
5) «Ход работы:» затем основной текст.
   - Писать по существу: что выполнено, какие решения приняты, какие проверки сделаны.
   - Если тема техническая: указывать только реально задействованные артефакты/узлы (точные пути, названия контроллеров/сервисов/контекстов БД), например:
     «API/Controllers/AuthController.cs», «Application/Services/AuthService.cs», «Infrastructure/Data/BankDbContext.cs».
   - Код внутрь текста не вставлять.
   - Не писать “инварианты” и общие декларации вида “структура остаётся неизменной”.
   - Не добавлять предположительных эффектов (“улучшит”, “повысит” и т.п.).
   - Если тема организационная (инструктаж/оргструктура): не выдумывать названия приказов, номера документов, ФИО сотрудников, внутренние регламенты. Писать нейтрально: «по предоставленным материалам/в установленном порядке/по схеме оргструктуры».
6) Скриншоты:
   - Используй маркер [ФОТО]. В одной зоне — один скрин.
   - Между любыми двумя зонами [ФОТО] обязателен как минимум один абзац текста.
   - Внутри [ФОТО] кратко укажи, что вставить (например: «Swagger: вызов POST /api/auth/login и ответ 200», «pgAdmin/Neon: список таблиц Users/Accounts/...», «логи запуска приложения», «файл AuthService.cs — метод генерации JWT»).
   - Сразу под зоной подпись строго: «Рисунок N — ...» (без точки).
   - Нумерация рисунков сквозная:
     • если я дал n — начать с n и далее +1;
     • если n не дал и нет контекста — начать с 1.
7) **Вывод:** 1–2 коротких предложения. Только факт выполненного результата (что сделано/настроено/подготовлено). Без оценок.

Оформление и стиль:
- Кавычки только «…».
- Жирным — только метки **Цель:**, **Задачи:**, **Вывод:** (и текст сразу после двоеточия/заголовка).
- Никакого курсива.
- Не раскрывать секреты: токены, пароли, ключи, полные connection strings. При необходимости используй <REDACTED>.
- Текст должен быть конкретным и проверяемым: только то, что относится к теме дня.
